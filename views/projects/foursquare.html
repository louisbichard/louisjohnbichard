1. Approach

Full end to end approach

Whilst there wasn't much time in this process, I wanted to really do a mini end to end build, incorporating as many aspects that I use in a full design process

My process for delivering this type of application / feature would not change, but given more time I would have spent a little more time on each of the phases of the design using different prototype methods and creating alternative mock interfaces and using tools such as A/B testing to work out the best designs.

I was conscious and didn't want to just 'dive into' the project. I wanted to spend the time thinking about the users, the user journeys etc. This was obviously difficult without having direct access to any customer as this is a mocked scenario, but lets pretend that I did, and stages such as the design sketches were put through iterations and review as would be standard in a larger process.

Lean methodology

It is important to me not to just dive into the build of such an application as this is a sure fire way to waste time if I don't have a clear idea of what I'm doing.

Instead, I started with the cheapest method: pen and pencil to sketch out ideas I had seen from other similiar applications, given more time this would have worked into other cheap methods of testing and validating hypothesis

3. Design

Data review

The first step was to review the data that was made available, this would shape the content of the design, was the API returning content in the form of images, text etc.

By asking these questions I was able to shape and form the rest of the design process in which the size / form of the data was important to how the interface was created.

Questions that were asked at this phase were:

  • Will we need to make subsequent requests to fetch further data, what API's would these be?
  • Which information is the most important and what should be given UI prominence?
  • Could all the information be displayed on a tile, or would expand / redirecting be required for extra information?
  • What call to actions would the user take?
  • Would the result set be large enough to require pagination?
  • Would the content need to be scrollable if there is too much of it?
  • What is the hierarchical structure of the data, is there a logical 'king' piece of content
  • What groupings would the content take?

What data do we have? Ratings? Images? What types of content are we delivering? How much data is there?

Existing design review

Look at other designs / UI mock ups that people have made for design inspiration. I identified what was liked / disliked about each design, and ideas that I wanted to incorporate into my own design



Sketching

Whilst looking at other ideas I would sketch different ideas that I had, so I wouldn't forget them, but also so that I could see on paper whether I liked the direction the design it was heading.




User journey considerations

Whilst this phase was not formally conducted, I did some thinking around the types of users that would enter the application and the types of journeys that these users were likely to take, this meant looking at other apps and trying to work out how they had innovatively solved some of these user Journeys.

Design process outcome

  • Really liked the map-on-the-right format for the UI, seemed to be consistent across major apps
  • Most apps didn't cater for different journeys / searching types, this would be a really delightful feature [2]

4. Technology choices

Bower

Bower is used to install app dependencies, keeping versioning clean and ensuring libraries can be updated easily.

Angular

I chose to build the app with Angular as this is a framework I've been working with a considerable amount in the past few years, so building flexible and clean architectures is a very quick process for me currently.

Not much of angular was leveraged in this project however if the app were to grow in the future, the beginning framework is already in place.

SASS

Sass is used for the styling, this helps to keep things neat within the component-based approach, all colours / shadows / transitions are controlled by single global variables meaning that the style guide of the architecture is flexible. Update one value for the colour and this will propogate to the rest of the app with a single change [6]

5. Build / Features



Performance

The search to the foursquare API [3] is debounced so that the user is given 500ms to stop typing before a request is made, giving the user time to enter their search term fully before getting their resultset.



Animations

On load the left panel animates in from the left

Component based approach

The interface is broken up into separate components, for instance the results is comprised of restaurant tiles, with this approach the content can be moved around VERY easily, keeping the architecture flexible and agile friendly [6]



Flexible style guide

As above the styles are created as isolated variables, meaning that all changes are propogated throughout the appliation keeping the architecture flexible and agile friendly [6]



6. Known defects

Drag / Click on map

Drag / Clicking is not currently working, as the z-index of the map container is affected in order to overlay the search, an alternative was not found within the deadline time

7. Improvements / Out of scope

Infinite scroll

The foursquare app has a "see more results" button at the bottom of their list, this is nice as it segments and paginates which helps the interface load fast, but the user has to click to see the results, an infinite scroll here would be much nicer and not require the user to have to click to see more results, simply scroll.

Typeahead

An autocomplete / typeahead would improve the UX when searching for items

Groupings / Filters

I wanted to really dive into the user journies and start to understand what types of tasks they were trying to achieve, one way that this would probably manifest is some clever filters or groupings that allow the user to quickly reduce the result sets into amounts that they require.

Alternative card based UI

Rather than map centric UI, a card based UI would allow the user to expand the search list to view extra details in a non location centric interface.

This was a feature that I identified as a nice-to-have in the design process when reviewing other sites.

Responsive UI

The left search panel is currently fixed width meaning smaller resolution devices do not get any meaningful UX
